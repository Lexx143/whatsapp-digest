# WhatsApp Daily Digest — полное описание для тиражирования

Инструмент читает локальную базу WhatsApp на Mac, расшифровывает голосовые, смотрит
картинки, анализирует переписку через Claude с долговременной памятью и присылает
структурированные сводки в Telegram-группу. Обучается на обратной связи прямо из чата.
Проверен в бою: ловит зависшие задачи с точностью до дня, пересечения отделов,
риски оттока клиентов; после внедрения контроля фактов — без ИИ-додумок.

## 1. Архитектура (конвейер)

```
launchd (пн–пт 09:30)
  └─ digest.sh  — оркестратор, вся логика по шагам:
      1. feedback.py   — забирает из Telegram ответы людей на сообщения бота → memory/feedback.md
      2. extract.py    — SQLite WhatsApp → JSON сообщений за окно «с прошлого успешного запуска»
      3. transcribe.py — голосовые .opus → текст (mlx-whisper), рендер transcript.md
      4. claude -p     — анализ (prompt.md): читает транскрипт, картинки, память; пишет дайджест + обновляет память
      4.5 claude -p    — КОНТРОЛЬ ФАКТОВ (prompt-verify.md): сверяет каждое утверждение с транскриптом, чистит неподтверждённое
      5. tg.py send    — отправка в Telegram-группу (разбивка >4096 симв.)
      6. сдвиг state/last_run — только после полного успеха и только вперёд
      7. по понедельникам — недельный обзор (prompt-weekly.md) вторым сообщением
      8. в первый понедельник месяца — месячный обзор (prompt-monthly.md) третьим
```

Любая ошибка шага → сообщение об ошибке в ту же Telegram-группу + лог в `logs/`.

Разделение кода и данных:
- код — этот репозиторий (постоянный путь!);
- данные — `~/.whatsapp-digest/`: `config.env` (токены), `memory/` (память и фидбэк),
  `digests/` (архив), `state/` (окно, offset, кэш расшифровок), `logs/`, `work/`, `venv/`.

## 2. Источник данных: база WhatsApp на macOS

Путь: `~/Library/Group Containers/group.net.whatsapp.WhatsApp.shared/`
- `ChatStorage.sqlite` — сообщения (обновляется, пока запущен WhatsApp Desktop);
- `ContactsV2.sqlite` — адресная книга;
- `Message/Media/...` — файлы медиа.

Ключевые знания о схеме (добыты методом проб — сэкономят вам день):
- Время: `ZMESSAGEDATE` — Core Data epoch, прибавляй **978307200** для unix.
- Чаты: `ZWACHATSESSION` (`ZCONTACTJID` — стабильный идентификатор группы/лички, задаётся в `common.py`).
- Имена в группах: `ZWAMESSAGE.ZGROUPMEMBER → ZWAGROUPMEMBER.ZMEMBERJID`, затем имя из
  ContactsV2 (`ZWAADDRESSBOOKCONTACT`: поля `ZLID` и `ZWHATSAPPID` → `ZFULLNAME`), фолбэк —
  `ZWAPROFILEPUSHNAME`. Поле `ZWAMESSAGE.ZPUSHNAME` — protobuf-мусор, НЕ использовать.
- Системные сообщения групп: `ZMESSAGETYPE=6`. Поле `ZGROUPEVENTTYPE` НЕ маркер
  системности (бывает =2 у обычных текстов) — не фильтровать по нему.
- Типы медиа: 1 изображение, 2 видео, 3 голосовое (.opus), 7 ссылка, 8 документ.
  Подпись к медиа — `ZWAMEDIAITEM.ZTITLE`, путь — `ZMEDIALOCALPATH` (относительно `Message/`).
- Читать базу ТОЛЬКО через копию (вместе с `-wal`/`-shm`) — extract.py так и делает,
  иначе конфликты с живым WhatsApp.

Окно анализа — «с прошлого успешного запуска» (`state/last_run`): пропущенные дни,
выключенный Mac и выходные подхватываются без дублей и дыр. launchd сам дозапускает
пропущенный по расписанию прогон после пробуждения Mac.

## 3. Расшифровка и картинки

- Голосовые: `mlx-whisper`, модель `mlx-community/whisper-large-v3-turbo` (Apple Silicon,
  русский — отлично). Кэш расшифровок по пути файла (`state/transcripts.json`) — один файл
  не гоняется дважды (важно: недельный/месячный обзоры переиспользуют дневные расшифровки).
- Картинки: не OCR-им заранее — Claude в headless-режиме читает файлы сам (инструмент Read
  понимает изображения). В транскрипте картинка = строка `[изображение, файл: /абс/путь]`,
  промпт велит читать выборочно только несущие сигнал.

## 4. Анализ: Claude CLI в headless

- `claude -p "<промпт>" --model sonnet --allowedTools "Read,Write"` из каталога данных.
  Работает по подписке Claude (отдельного API-ключа не нужно), но CLI надо один раз
  авторизовать: `claude` → `/login` в терминале (логин десктоп-приложения НЕ наследуется).
- Дневному прогону разрешён Write (обновляет память), недельному/месячному и контролёру — только Read.
- Грабли: если запускаете claude -p из-под другой Claude-сессии — сначала
  `unset ANTHROPIC_BASE_URL CLAUDECODE CLAUDE_CODE_*`, иначе «Not logged in».
- stdout прогона = дайджест; всё до первого эмодзи-заголовка отрезается кодом.

## 5. Обучаемость — два контура

**Долговременная память** (`memory/context.md`) — ведёт сам дневной прогон по инструкции
из prompt.md: клиенты / задачи в работе / риски / архив, каждая сущность с датами.
Поэтому бот говорит «клиент N — 14-й день без движения» и замечает, что задача «молча» исчезла
из планов. Недельный и месячный обзоры читают память для динамики, но не пишут в неё
(один владелец у файла — нет конфликтов).

**Обратная связь** (`memory/feedback.md`) — любой участник Telegram-группы отвечает
(reply) на сообщение бота или упоминает его @username; перед каждым прогоном feedback.py
забирает это через getUpdates (offset в `state/tg_offset`) и дописывает в файл с датой
и автором. Промпты велят применять эти указания приоритетнее собственных представлений,
свежие — важнее старых. Так бот скорректировал: что считать важным, чьи имена как
трактовать, фактические поправки. Расписание фидбэком НЕ меняется — только код/launchd.

## 6. Достоверность (анти-«ИИ-слоп»)

После жалобы команды на додумки введены два механизма:
1. **Маркировка факт/вывод** (prompt.md): утверждение без пометки = дословно из переписки;
   интерпретации обязаны начинаться с «вывод:».
2. **Контролёр фактов** (prompt-verify.md): отдельный проход перед КАЖДОЙ отправкой —
   сверяет каждое утверждение дайджеста с транскриптом, неподтверждённое исправляет или
   удаляет, ничего не добавляет. На первом же прогоне поймал 2 реальные додумки.

## 7. Три уровня отчётности (заказ генерального)

- **Дневной** (пн–пт 09:30): короткий, читается за 30 секунд. Блоки: 🚨 критическое /
  🔀 между отделами (приоритет №1) / 👥 менеджеры одной строкой с пометкой
  (клиент ✓ | внутреннее | риск) / ⚠️ план-факт и эскалации зависшего.
- **Недельный** (пн, второе сообщение): вехи, повторяющиеся заявки/проблемы, динамика
  очередей и план/факт-дисциплины, итоги + перенос на следующую неделю.
- **Месячный** (первый пн месяца, третье сообщение): любая измеримая динамика
  (клиенты/очереди/дисциплина/время жизни задач/повторные заявки), куда направить
  внимание, + «гигиена отчётов» — мета-анализ, какие элементы отчётов информативны,
  а какие шум.

Полные тексты промптов — в файлах `prompt.md`, `prompt-weekly.md`, `prompt-monthly.md`,
`prompt-verify.md` этого репозитория. Это главная «бизнес-логика» инструмента — правьте
их под свой отдел, код трогать не придётся.

## 8. Установка с нуля (~30 минут + скачивание модели)

1. Mac с WhatsApp Desktop (залогинен и синхронизирован) — это единственная платформа,
   где база лежит в открытом виде. Windows-клиент шифрует базу — там нужен другой подход.
2. `git clone` в постоянную папку (НЕ временную! планировщик запоминает путь).
3. Найти JID своих чатов и вписать в `~/.whatsapp-digest/chats.json` (`{"<jid>": "Название"}`):
   `sqlite3 "file:$HOME/Library/Group Containers/group.net.whatsapp.WhatsApp.shared/ChatStorage.sqlite?mode=ro" "SELECT ZCONTACTJID, ZPARTNERNAME FROM ZWACHATSESSION WHERE ZPARTNERNAME LIKE '%ИмяЧата%';"`
4. `claude` → `/login` (разово).
5. `./install.sh` — создаст `~/.whatsapp-digest/`, venv c mlx-whisper (~1.5 ГБ модель
   при первом прогоне), LaunchAgent пн–пт 09:30.
6. Бот: @BotFather → `/newbot` → токен в `~/.whatsapp-digest/config.env`; создать
   Telegram-группу, добавить бота, написать что-нибудь, `python3 tg.py chat-id` → chat_id
   в config.env.
7. Тест: `./digest.sh` (за период: `./digest.sh --from 2026-07-01`; недельный вручную:
   `WA_WEEKLY=1 ./digest.sh`; месячный: `WA_MONTHLY=1 ./digest.sh`).

## 9. Известные ограничения

- Mac должен быть включён, WhatsApp Desktop — запущен (иначе база не обновляется).
- Видео не расшифровываются (помечаются с подписью).
- Обратная связь применяется со следующего прогона, бот не отвечает в чате мгновенно.
- Пустое окно сообщений → бот молчит (осознанно, чтобы не слать «сводку ни о чём»).
- WhatsApp-групп с ботами не бывает: читаем локальную базу, пишем в Telegram. Полный
  переезд в облако возможен только через неофициальное «связанное устройство»
  (Baileys) с риском бана номера — решение отложено сознательно.

## 10. Куда развивать (проверено экспериментом)

Интеграция с ASISTA CRM: `/tasks/list` по пресетам даёт машинные цифры очередей,
приток/отток заявок, повторные заявки — вместо скриншотов в чатах. Грабли: работает
только по пресетам залогиненного аккаунта, `finished_at` = SLA-дедлайн (не факт
закрытия — брать дату статуса «Завершено»), refresh-токен ротируется раз в ~7 дней.
